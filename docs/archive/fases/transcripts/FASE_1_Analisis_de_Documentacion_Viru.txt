1.5 Propuestas tecnicas para cerrar lagunas (alternativas razonadas)
Aqui no decido aun "la arquitectura final" (eso es FASE 2), pero si propongo opciones de solucion para cada hueco, con trade-offs claros, para que tu como PO puedas escoger.
1.5.1 Definicion de "precio trackeado" (propuesta contractual)
Opcion A - Precio minimo base "desde" (recomendado para MVP)
Trackear el precio mas bajo que Ryanair muestre para 1 adulto sin extras, en la moneda del mercado del origen.
Ventajas: simple, comparable, mas estable para historico.
Inconvenientes: el usuario final puede pagar mas en checkout; hay que explicar "precio orientativo".
Opcion B - Precio final parametrizable
Permitir configurar: numero de pasajeros, equipaje, tarifa, etc.
Ventajas: mas exacto para el usuario.
Inconvenientes: mucha complejidad, mas fragil ante cambios de Ryanair, mas coste.
Opcion C - Doble precio: base + estimacion final
Guardar precio base y un "estimado final" (por reglas).
Ventajas: mejor UX sin scraping extremo.
Inconvenientes: requiere modelado de reglas y disclaimers.
Recomendacion inicial para coherencia con documentacion: A para historico/alertas, con disclaimers y futura ampliacion a B/C como premium.
1.5.2 Identidad del vuelo: dia vs hora
Opcion A - Track por "ruta + fecha" (dia)
En el tracker: para ese dia, guardar el "best price" observado (o el precio del vuelo mas barato del dia).
Calendario encaja perfecto.
Contras: no distingues horarios; un "vuelo" puede "cambiar" si cambia cual es el mas barato.
Opcion B - Track por "ruta + fecha + salida (hora)"
Implica pedir hora/seleccionar un vuelo concreto al anadir.
Pro: historico mas fiel; alertas precisas.
Contra: UX mas pesada; calendario requiere agregacion.
Opcion C - Ambos (modelo jerarquico)
"RouteDay" como entidad y "Departures" como subentidades.
Pro: soporta ambos UIs.
Contra: mas costoso, pero escalable.
Dado que el esquema actual solo pide "fecha", la opcion mas alineada es A, pero para el futuro (comparativas y escalas) lo mas robusto suele ser C.
1.5.3 Frecuencia de escaneo y "momentos clave"
Opcion A - Cron fijo configurable (ej. 3-6 veces/dia)
Parametrizable por entorno (dev/stage/prod) y por plan (free/premium).
Anadir jitter para evitar patrones.
Opcion B - Event-driven adaptativo
Subir frecuencia cuando el vuelo se acerca o cuando detectas volatilidad.
Mas valor al usuario, menos coste global, pero mas complejidad.
Opcion C - Hibrido
Base fija + refuerzos por reglas (si usuario pide "cada cambio", si objetivo cercano, etc.).
La documentacion sugiere "momentos clave" -> C encaja mejor (base + refuerzo), especialmente si existe "alerta por cada cambio".
1.5.4 Notificaciones: definicion de "email o notificacion"
Opcion A - Email + notificacion in-app
Notificacion = campana/bandeja interna, ademas del email.
Pros: facil en web, no requiere push.
Contras: menos inmediata.
Opcion B - Email + Web Push
Push del navegador para alertas rapidas.
Contras: permisos, compatibilidad.
Opcion C - Email + movil (roadmap)
Mantener web ahora, anadir app movil luego.
Por coste/beneficio y ambiguedad del doc, A es la opcion segura para MVP, con camino directo a B.
1.5.5 "IA" de busqueda rapida vs heuristica
Para evitar prometer "IA" donde hay heuristica:
Renombrar internamente a Search Optimizer (heuristico) y reservar "IA" para:
prediccion de precios,
ranking inteligente de combinaciones,
o recomendador.
A nivel de producto, puedes seguir llamandolo "IA" si quieres, pero tecnicamente propondria:
MVP: heuristica (radio, fechas flexibles, filtros).
Fase ML: ranking con features (precio historico, distancia, horarios preferidos).
1.5.6 Escalas/self-connect: reglas minimas y disclaimers
Para cumplir "advertir que puede ir justo" de forma implementable:
Definir un MCT configurado (ej. 90-150 min) por aeropuerto (si no hay datos, usar default).
Senalar:
"self-connect no protegido",
"si pierdes el segundo, es tu responsabilidad",
"equipaje y controles pueden requerir re-check".
Y en UI:
mostrar buffer (minutos) y "riesgo: bajo/medio/alto" por heuristica.
Esto aterriza el requisito de advertencia sin necesidad de datos perfectos.

1.6 Riesgos principales ya visibles y mitigaciones iniciales (sin entrar aun en DevOps)
1.6.1 Dependencia de Ryanair (cambios, bloqueos, legal)
Riesgo: si el sistema obtiene precios de manera automatizada, puede romperse por cambios de UI o medidas anti-bot.
Mitigacion: "Provider abstraction", cache, backoff, rotacion, y modo degradado (mostrar ultima muestra).
Mitigacion de producto: transparencia "ultima actualizacion: timestamp" y control de frecuencia.
(No "decido" el metodo aun; esto es FASE 2/3, pero el riesgo se documenta desde ya.)
1.6.2 Experiencia de usuario vs ruido (alertas)
Riesgo: alertas cada cambio = spam.
Mitigacion: throttling, digest, umbrales minimos de variacion, y configuracion por vuelo.
1.6.3 IA: expectativas y responsabilidad
Riesgo: el usuario interpreta "compra ahora" como garantia.
Mitigacion: disclaimers, probabilidad/confianza, y lenguaje cuidadoso.
1.6.4 Datos incompletos para escalas
Riesgo: recomendar conexiones imposibles o muy arriesgadas.
Mitigacion: MCT conservador, bloqueos por overnight si no se soporta, etiquetado de riesgo.

Esto deja un hueco para una herramienta especializada centrada en Ryanair. Ryanair sigue siendo la aerolinea low-cost con mayor volumen en Europa, por lo que existe demanda real de usuarios que buscan seguimiento y alertas. Este contexto justifica timing y foco del producto.

El resumen del proyecto enumera lineas de monetizacion y escalabilidad a futuro:
- Modelo freemium o suscripcion: base gratuita con funciones premium (alertas ilimitadas, canales avanzados tipo SMS/WhatsApp, estadisticas, prediccion mas precisa, experiencia sin anuncios).
- Comisiones de afiliado por reservas: deep-link a Ryanair con seguimiento de conversion para comisiones.
- Publicidad segmentada con moderacion: ofertas de hoteles, seguros, alquiler de coches, sin afectar UX.
- Expansion a otras low-cost: Wizz Air, easyJet, Vueling, etc., creando un ecosistema de trackers.
- Crecimiento de base de usuarios y datos: comunidades, compartir chollos, y posible uso de datos agregados para reportes B2B.

1.10 Ampliacion de ideas (nuevas propuestas de producto)
- Modo "Objetivo de ahorro": el usuario define cuanto quiere ahorrar y el sistema sugiere rutas/fechas con mayor probabilidad de bajar a ese nivel.
- Alerta de estabilidad: avisar cuando un precio se mantiene estable durante X dias para reducir ansiedad y evitar compras impulsivas.
- Indicador de volatilidad: etiqueta simple (baja/media/alta) basada en cambios recientes para guiar decision.
- Comparador de ventanas: mostrar 3 ventanas cercanas (por ejemplo +/- 3 dias) con mejor precio para decision rapida.
- Modo "viaje flexible": si el usuario no fija destino, se muestran los destinos con mayor caida porcentual.
- Alertas por meta de fecha: notificar si el vuelo baja dentro de una ventana temporal concreta (ej. antes de 10 dias).
1.11 Matriz de valor vs esfuerzo (sugerencia)
Clasificar cada mejora por: valor para usuario, complejidad tecnica, riesgo de proveedor. Esto ayuda a priorizar sin depender de intuicion.
1.12 Lista de supuestos criticos a validar temprano
- Que el proveedor permita consultas sin bloqueos graves.
- Que el usuario entienda el precio trackeado como orientativo.
- Que los historicos sean utiles sin saturar la UI.
- Que las alertas generen accion real (click a compra).
1.13 Ideas avanzadas (estrategia y crecimiento)
- Segmentacion de usuarios: viajeros frecuentes vs viajeros ocasionales; adaptar alertas y lenguaje.
- KPI de valor: porcentaje de usuarios que ahorran X euros en su primer mes.
- Experimento temprano: medir si el usuario vuelve a consultar tras recibir una alerta.
- Modo "objetivo por viaje": el usuario define un evento (boda, congreso) y se priorizan alertas.
- Escalado por ciudades: empezar por rutas con mayor volumen Ryanair.
1.14 Reglas de coherencia global (aplicadas)
- Busqueda rapida = heuristica explicable en MVP. IA real se reserva para prediccion de precios.
- Precio trackeado en MVP = precio base para 1 adulto sin extras (orientativo).
- Identidad de vuelo en MVP = Ruta + Dia. Hora opcional en evolucion.
- Canales de alerta en MVP = email + bandeja in-app. Web push es mejora futura.
1.15 Metricas minimas globales (lista canonica)
- Porcentaje de vuelos con datos frescos.
- Ratio de alertas entregadas vs abiertas.
- Conversion alerta -> click a Ryanair.
- Tiempo medio entre alerta y compra (si hay tracking de afiliacion).
