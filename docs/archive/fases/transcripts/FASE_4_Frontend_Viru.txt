Viru - Ryanair Tracker
Documento tecnico - Arquitectura frontend, UX/UI, flujos, estado y validaciones
Fecha: 2026-02-12
0. Objetivo y alcance de la Fase 4
Esta fase define la arquitectura frontend de Viru con nivel enterprise y continuidad directa con la Fase 2 de arquitectura global y la Fase 3 de backend en FastAPI. El objetivo no es solo describir pantallas, sino especificar contratos de integracion, comportamiento en condiciones reales, estados de interfaz, reglas de validacion y decisiones de experiencia de usuario que reduzcan errores y friccion.
El alcance incluye la aplicacion web completa para usuarios autenticados: registro y login, panel principal, alta y gestion de vuelos trackeados, historico de precios en grafico y calendario, busqueda rapida con alternativos y escalas opcionales, alertas por umbral o por cambio, preferencias guardadas, sugerencias de usuario e internacionalizacion ES/EN. Tambien se cubren criterios de accesibilidad, rendimiento, observabilidad del cliente y protecciones de seguridad en la capa browser.
Este documento toma como fuente funcional principal los documentos de producto previos y respeta las decisiones de backend ya establecidas: API versionada en /api/v1, jobs asincronos para tracking, proveedor Ryanair desacoplado mediante adaptador con ryanair-py en servidor, y mecanismo de degradacion controlada cuando la fuente externa no responde. El frontend, por diseno, no consulta fuentes externas de vuelos directamente; toda consulta de negocio pasa por backend para mantener coherencia, control de seguridad y trazabilidad.
4.1 Stack frontend y principios de implementacion
Decision D-FE-001: se adopta Next.js con React y TypeScript como base del frontend. Esta eleccion permite combinar rutas con renderizado en servidor para contenido publico y renderizado en cliente para el panel autenticado con alta interaccion. TypeScript se usa de forma estricta para minimizar errores de integracion y mantener contratos estables con el backend.
Decision D-FE-002: la arquitectura de estado se separa en server state y client state. El server state (datos remotos, cache, invalidacion, reintentos) se gestiona con TanStack Query. El client state efimero (filtros de pantalla, modal abierto, pestaña activa, preferencias temporales no persistidas) se gestiona con estado local y stores ligeros por dominio. Esta separacion evita anti patrones de usar un store global para todo.
Decision D-FE-003: formularios y validaciones se implementan con React Hook Form y Zod. El contrato de cada formulario se alinea con los DTO de backend para impedir drift semantico entre cliente y servidor. Las validaciones de frontend nunca sustituyen validacion de servidor; su funcion es mejorar UX al dar feedback temprano.
Decision D-FE-004: sistema de estilos con Tailwind CSS + libreria de componentes base (shadcn/ui o equivalente), complementado por un sistema de tokens de diseno. El uso de tokens permite consistencia visual y reduce deuda en temas de color, tipografia, espaciado y estados de interaccion.
Decision D-FE-005: observabilidad de frontend desde el dia uno. Se instrumentan errores JavaScript, errores de red, tiempos de navegacion y eventos funcionales criticos como alta de vuelo, refresh manual y activacion de alertas. Sin estas metricas, la mejora de conversion y retencion se vuelve especulativa.
4.2 Arquitectura frontend detallada
La arquitectura del cliente se diseña por vertical slicing: cada dominio funcional mantiene su carpeta de componentes, hooks, modelos de vista y servicios de integracion. Esto reduce acoplamiento entre pantallas y mejora la velocidad de cambios. Los elementos compartidos se concentran en una capa comun estable: design system, utilidades de formateo, cliente HTTP, i18n y telemetria.
Se implementa un BFF ligero en Next.js solo para necesidades de borde muy concretas: proteccion de rutas inicial, paso de cabeceras de sesion en SSR y prerender de contenido no sensible. La logica de negocio permanece en FastAPI para no duplicar reglas en dos capas. Cualquier decision de calculo funcional debe residir en backend y exponerse por API versionada.
El frontend asume un modelo de comunicacion resiliente: timeouts explicitos, retries controlados solo donde tengan sentido, y fallback visual cuando el backend entra en modo degradado por proveedor externo. El usuario siempre ve estado verificable: datos actuales, fecha de ultima observacion, y motivo de ausencia de nuevos datos cuando aplique.
La navegacion se estructura en rutas publicas y privadas. Publicas: landing, login, registro, politicas. Privadas: dashboard, watchlist, estado de vuelos, historico, busqueda rapida, alertas, preferencias y sugerencias. El guardado de acceso se implementa con middleware y comprobacion de sesion en servidor para evitar parpadeo de contenido privado.

4.2.2 Contratos de integracion con backend
El frontend consume exclusivamente endpoints /api/v1 definidos en backend. Se implementa un cliente HTTP con interceptores para token, correlacion de trazas y manejo uniforme de errores. Cada endpoint tiene su esquema de respuesta tipado y validado en runtime para detectar cambios no compatibles antes de romper la UI.
Se usa estrategia contract first en cliente: para cada recurso se define request, response, mapping a view model y politicas de cache. Esto evita exponer la forma cruda del backend en componentes visuales. Los componentes trabajan con view models orientados a UX, por ejemplo PriceTrendCardModel o WatchStatusBadgeModel.
Para operaciones criticas como refresh manual o alta de vuelo se usan idempotency keys desde cliente, de modo que el usuario no provoque duplicidad por doble click o reconexion inestable. En caso de 202 Accepted, el frontend muestra estado pending y hace polling progresivo con corte por timeout razonable.
Errores de autorizacion se tratan con cierre de sesion seguro y redireccion a login preservando returnUrl. Errores 429 muestran mensaje de limite de frecuencia y sugerencia de reintento. Errores 5xx muestran fallback no destructivo con CTA para reintentar y acceso al ultimo dato confirmado.
4.3 Diseño UX/UI y sistema de componentes
La experiencia de usuario se diseña para dos perfiles principales: usuario que monitoriza 1-3 rutas concretas y usuario intensivo que compara rutas, fechas y alertas de forma diaria. El primer perfil necesita rapidez y claridad. El segundo necesita densidad informativa, filtros y comparativas sin perder legibilidad.
Se define una jerarquia visual estricta: cabecera con contexto de ruta o seccion, bloques de metrica resumida, contenido principal exploratorio y acciones secundarias en panel lateral o menu contextual. Este orden reduce sobrecarga cognitiva y evita que acciones de alto riesgo, como borrar seguimiento, queden cerca de acciones rutinarias.
El sistema de componentes sigue principios de composicion: botones, inputs, selectores de fecha, chips de filtro, tablas, tarjetas de vuelo, grafico temporal, calendario de precios, banners de estado y toasts transitorios. Cada componente incluye variantes de estado: default, loading, empty, error y disabled.
Las decisiones de microinteraccion se documentan: skeleton en carga inicial, spinner solo en acciones puntuales, transiciones cortas para cambios de modo grafico/calendario y confirmaciones explicitas en acciones destructivas. Se evita animacion decorativa excesiva para mantener rendimiento en dispositivos de gama media.
4.3.1 Tokens de diseno, consistencia y theming
Se define un set de tokens para color, tipografia, radio, sombras y espaciado. Los tokens se exportan como variables semanticas y no como valores hardcoded en componentes. Esto permite evolucionar branding sin refactor masivo.
El modo de color inicial puede ser claro con soporte posterior a tema oscuro. Aunque el proyecto no exige dark mode en esta fase, se recomienda preparar contraste y semantica de color desde el inicio para no crear deuda de accesibilidad.
Los colores de estado se reservan con semantica estable: exito para minimos favorables, alerta para subidas abruptas, informativo para ultima actualizacion y neutral para estados pendientes. En calendario se aplican convenciones consistentes para min y max evitando depender solo de color; se incluyen etiquetas y tooltip textual.
La escala tipografica diferencia titulos operativos, datos de precio y texto de ayuda. Los precios y variaciones deben destacar visualmente sin distorsionar el layout en pantallas moviles. Se definen truncados y line-clamp para nombres largos de aeropuerto o rutas compuestas.

4.4 Flujos de usuario y comportamiento de interfaz
Los flujos se disenan como maquinas de estado para evitar ambiguedad en transiciones. Cada pantalla define estados primarios, eventos de usuario y efectos esperados. Esta disciplina reduce bugs de UI donde el estado visual no coincide con el estado real de negocio.
Todos los flujos de escritura incluyen confirmacion visible de resultado: exito, pendiente o error con detalle accionable. Nunca se cierra un formulario critico sin feedback. Los mensajes se redactan con tono operativo y no ambiguo, alineado con la prioridad de confianza que requiere un producto de seguimiento de precios.
La navegacion conserva contexto. Si el usuario viene de historico comparativo y abre detalle de un vuelo, al volver recupera filtros y posicion. Esta persistencia de contexto se implementa con query params y estado serializable, no con memoria opaca del navegador.
4.4.1 Flujo de registro y login
Registro: el usuario completa email, password y aceptacion de terminos. La validacion en cliente verifica formato y fortaleza basica; el servidor aplica politicas definitivas. Tras registro exitoso se puede requerir verificacion de email segun politica de seguridad y antispam.
Login: el usuario envía credenciales y obtiene sesion. Si hay error de autenticacion, el mensaje evita filtrar informacion sensible. Si la cuenta existe pero esta bloqueada temporalmente por intentos fallidos, se informa ventana de espera y canal de soporte.
Recuperacion de sesion: ante token expirado en una ruta privada, la app intenta refresh seguro una sola vez. Si falla, cierra sesion de forma limpia y redirige a login. El usuario conserva returnUrl para reanudar flujo tras autenticarse.
4.4.2 Flujo de alta de vuelo trackeado
Desde watchlist, el usuario abre AddWatchForm y completa origen IATA, destino IATA, fecha y precio objetivo opcional. Se valida que origen y destino no sean iguales, fecha no pasada y formato correcto de codigos. El sistema sugiere alternativas cuando el codigo no coincide con aeropuerto valido.
Al enviar, el frontend crea comando de alta con idempotency key y espera confirmacion. Si el backend responde accepted por trabajo asincrono inicial, la UI muestra estado provisional y marca el nuevo item como pending-first-scan. Cuando llega la primera observacion, el estado pasa a active.
Si existe ya una ruta equivalente segun la regla de unicidad vigente, la UI ofrece dos caminos: abrir el seguimiento existente o crear variante si la politica lo permite. Esta decision evita duplicidad involuntaria y mantiene limpieza en watchlist.
4.4.3 Flujo de historico: grafico, calendario y comparativa
El historico ofrece conmutador entre modo grafico y modo calendario. El cambio de modo preserva filtros de rango temporal y vuelo seleccionado. En grafico, cada punto expone tooltip con precio, fecha, hora y fuente. En calendario, cada dia muestra precio representativo y estado cromatico min/max.
Comparativa: el usuario puede seleccionar dos o mas vuelos y superponer series o abrir vista lado a lado segun densidad de datos. Para evitar ruido, la UI limita comparativa inicial a un maximo razonable con opcion de ampliar bajo demanda. Se incluyen leyendas claras y control de visibilidad por serie.
Estados vacios: si un vuelo aun no tiene observaciones, se muestra empty state util con explicacion y CTA de refresh manual. Si hay datos parciales por degradacion de proveedor, se muestra banner de transparencia con fecha de ultimo dato confirmado.
4.4.4 Flujo de busqueda rapida con alternativos y escalas
El formulario de busqueda rapida recibe origen, destino, fecha y parametros de flexibilidad: radio de aeropuertos alternativos y margen de dias. Tambien incluye opcion de escalas. La interfaz explica que escalas implican self-connect y pueden aumentar riesgo operativo.
En resultados se muestran tarjetas o tabla con precio, distancia al aeropuerto alternativo, ventanas horarias y nivel de riesgo de conexion cuando haya escala. El ranking inicial prioriza equilibrio entre precio y viabilidad, no solo minimo absoluto.
El usuario puede aplicar filtros avanzados: horario de salida, duracion maxima total, exclusion de aeropuertos y rango de precio. Cada cambio de filtro actualiza resultados sin recargar pagina y conserva una referencia de consulta para trazabilidad y comparticion interna.
4.4.5 Flujo de alertas, canales y preferencias
Desde la vista de alertas, el usuario define reglas por vuelo: umbral de bajada, subida abrupta o notificar cada cambio. La UI muestra claramente implicaciones de volumen de avisos. Para modos intensivos se recomienda cooldown para reducir ruido.
Canales: email y notificacion in-app como base de MVP. Si un canal no esta verificado o no disponible, el formulario lo refleja y sugiere pasos de activacion. Cada alerta guardada muestra ultima evaluacion y ultimo envio para facilitar diagnostico.
Preferencias globales: idioma, radio por defecto, exclusiones horarias y aeropuertos a evitar. Estas preferencias precargan formularios y mejoran velocidad operativa. Cada cambio de preferencias se persiste con confirmacion y versionado para auditoria.
4.5 Estado de aplicacion, cache y validaciones
El modelo de estado se define para evitar inconsistencias entre varias pantallas que comparten datos de vuelos y observaciones. La fuente de verdad para datos remotos es TanStack Query con keys por dominio y parametros de consulta. La invalidacion se hace por eventos de mutacion y no por refresh global indiscriminado.
Las politicas de cache distinguen recursos estaticos, semiestaticos y volatiles. Preferencias e i18n tienen TTL largo. Watchlist y estado de vuelos tienen TTL medio con refetch en foco. Historico y resultados de busqueda usan cache contextual con caducidad corta y controles de stale data.
En formularios se usa validacion incremental: cliente para formato y ergonomia, servidor para reglas de negocio. Los mensajes de error se asocian al campo y a un resumen superior cuando hay varios fallos. Se evita mostrar tecnicismos de backend al usuario final.
4.5.1 Estrategia de server state
Query keys se disenan con granularidad suficiente para evitar invalidaciones masivas. Ejemplo: ['watch', watchId, 'history', range] frente a una key unica de historico global. Esta granularidad reduce peticiones y mejora responsividad.
Mutaciones criticas aplican optimistic UI solo cuando la probabilidad de conflicto es baja. Para alta de vuelo y cambios de alerta se puede aplicar optimistic parcial con rollback visible. Para operaciones dependientes de job asincrono se prefiere pending explícito en lugar de optimismo total.
El polling se usa de forma limitada en estados pending. Se implementa backoff progresivo y corte temporal para no saturar servidor ni navegador. Cuando el job termina, se invalida solo el subconjunto de queries afectadas.
4.5.2 Estrategia de client state
El client state incluye filtros, ordenacion, modal state y estado de UI de componentes avanzados. Se mantiene local por pagina y se eleva solo cuando hay verdadera necesidad de compartirlo entre rutas.
Para preservar experiencia al navegar, parte del estado se serializa en query params: rango de fechas, modo de vista, vuelos comparados y filtros de resultados. Esto facilita deep-linking y reproducibilidad de incidencias.
Se evita almacenar datos sensibles en localStorage. Para preferencias no sensibles se usa persistencia controlada con versionado de schema y limpieza en logout.
4.5.3 Formularios y contratos de validacion
Cada formulario tiene un schema Zod que refleja restricciones de dominio: IATA de 3 letras, fecha valida, umbrales numericos positivos, y limites de rango razonables. Estos esquemas se reutilizan para tipado de componentes y para tests de regresion.
Las validaciones cross-field son criticas. Ejemplo: fecha de regreso no puede ser anterior a ida cuando aplique; hora minima de salida no puede superar hora maxima; umbral de subida abrupta debe tener minimo tecnico para evitar alertas espurias.
Mensajeria de error se localiza en ES y EN con claves estables. Cada error tiene version corta para inline y version larga para ayuda contextual. Este detalle mejora conversion en formularios complejos como busqueda rapida.
En envios repetidos por doble click, el boton entra en estado loading y queda deshabilitado hasta respuesta o timeout controlado. Si el timeout supera umbral, se ofrece opcion de reintentar sin duplicar operacion mediante idempotency key.
4.5.4 Manejo de errores y resiliencia de UX
Se define taxonomia de errores en frontend: validacion, autenticacion, autorizacion, limite de frecuencia, dependencia externa degradada, y error interno no clasificado. Cada clase tiene patron visual y accion recomendada.
Cuando backend reporta degradacion del proveedor de vuelos, la UI no colapsa. Muestra ultimo dato consistente, sello temporal y advertencia de que el valor puede no reflejar cambio reciente. Esta transparencia evita perdida de confianza.
Para errores no recuperables, se registra evento de telemetria con hash de contexto y se ofrece al usuario un identificador de incidencia. Esto acelera soporte tecnico sin exponer informacion sensible.
4.6 Internacionalizacion, accesibilidad y contenido contextual
La internacionalizacion cubre no solo textos de interfaz, tambien plantillas de notificacion, validaciones, mensajes de alertas y textos de ayuda en busqueda con escalas. El sistema usa diccionarios por namespace y fallback definido para claves faltantes.
Accesibilidad: cumplimiento objetivo WCAG 2.2 AA en componentes criticos. Se exige navegacion completa por teclado, foco visible, contraste suficiente, etiquetas ARIA cuando corresponda y estructura semantica correcta en tablas y formularios.
Los graficos de historico deben ofrecer alternativa textual para usuarios con lectores de pantalla y para exportes. Las tendencias principales y min/max se resumen en texto junto al grafico para no depender exclusivamente de representacion visual.
El modulo de consejos por pais se presenta como contenido orientativo con version y fecha de revision visibles. El frontend debe mostrar claramente que esa informacion no sustituye confirmacion oficial por parte del usuario.
4.7 Rendimiento, observabilidad y seguridad en cliente
Rendimiento: objetivo de experiencia fluida en dispositivos medios con red movil. Se aplica code-splitting por ruta y carga diferida de componentes pesados como graficos. Las dependencias de visualizacion se cargan on demand para no penalizar login y dashboard.
Se optimiza Core Web Vitals con estrategia de prefetch selectivo, minimizacion de JS en rutas publicas y uso de imagenes optimizadas. Las tablas extensas en historico o resultados usan virtualizacion cuando el volumen supera umbral definido.
Observabilidad frontend: se capturan metricas de navegacion, latencia de acciones, errores de red y puntos de abandono de formulario. Los eventos incluyen correlation_id para enlazar con trazas backend y diagnosticar problemas end to end.
Seguridad cliente: CSP estricta, proteccion frente a XSS por sanitizacion de contenido dinamico, politica de cookies seguras para sesion, y reduccion de superficie de datos en navegador. El frontend nunca expone secretos, tokens de proveedor ni reglas internas sensibles.
Se añade proteccion anti automatizacion en endpoints de alta sensibilidad a traves de backend y señales de cliente. En frontend se limita informacion detallada de errores de autenticacion para no facilitar enumeracion de cuentas.
4.8 Edge cases frontend, riesgos y mitigaciones
Edge case E-FE-001: usuario cambia de idioma con formulario en progreso. Riesgo: invalidar estado y perder campos. Mitigacion: conservar valores del formulario y revalidar contra el mismo schema tras cambio de locale.
Edge case E-FE-002: watchlist con gran volumen de rutas y render lento. Mitigacion: paginacion o virtualizacion, memoizacion de filas y separacion de celdas costosas en componentes puros.
Edge case E-FE-003: resultados de busqueda con combinaciones de escalas de baja viabilidad. Mitigacion: filtros de riesgo por defecto, badges de advertencia y ocultacion inicial de conexiones con buffer insuficiente salvo accion explicita del usuario.
Edge case E-FE-004: respuestas inconsistentes por latencia entre refresh manual y escaneo programado. Mitigacion: ordenamiento por timestamp de observacion, version de snapshot y reconciliacion determinista en cliente.
Edge case E-FE-005: notificaciones frecuentes generan fatiga. Mitigacion: UI de resumen diario opcional, cooldown configurable y panel claro de ultima notificacion por regla.
Edge case E-FE-006: perdida de conectividad durante envio de formulario. Mitigacion: deteccion online/offline, cola local temporal para reintento seguro en acciones permitidas y feedback inmediato sobre estado de red.
Edge case E-FE-007: aeropuertos con codigos validos pero no soportados por proveedor en una ventana concreta. Mitigacion: mensaje explicativo, sugerencia de alternativos y conservacion de parametros para ajustar sin reiniciar formulario.
Edge case E-FE-008: usuario abre multiples pestañas y pisa configuraciones. Mitigacion: timestamp de ultima edicion, aviso de conflicto y preferencia por ultima escritura confirmada con notificacion contextual.
4.9 Plan de implementacion frontend por iteraciones
Iteracion 1: cimientos de plataforma. Incluye setup Next.js, autenticacion, layout base, routing privado, cliente HTTP tipado, i18n base y observabilidad minima. Criterio de salida: login y dashboard funcionales con sesion segura.
Iteracion 2: watchlist y estado de vuelos. Implementa alta de vuelo, tabla de seguimientos, estados activo/vencido, refresh manual y feedback transaccional. Criterio de salida: usuario puede crear y gestionar seguimiento extremo a extremo.
Iteracion 3: historico avanzado. Integra grafico, calendario, min/max, comparativa de vuelos y manejo de estados vacios o degradados. Criterio de salida: analitica visual util y estable en datos reales.
Iteracion 4: busqueda rapida y escalas. Incorpora alternativos, flexibilidad de fechas, filtros y etiquetas de riesgo de self-connect. Criterio de salida: flujo completo de exploracion y redireccion a compra.
Iteracion 5: alertas, preferencias y sugerencias. Incluye reglas, canales, ajustes globales y formulario de feedback. Criterio de salida: producto usable de manera recurrente con personalizacion real.
Iteracion 6: hardening. Enfocada en accesibilidad AA, performance, telemetria avanzada, tratamiento de edge cases y cierre de deuda UX detectada en pruebas.

4.11 Definition of Done de Fase 4
Todas las rutas privadas requieren sesion valida y controlan expiracion sin fuga de datos.
Formularios criticos (alta de vuelo, busqueda, alertas, preferencias) tienen schema tipado, validacion UI y pruebas.
Historico en modo grafico y calendario soporta estados loading, empty, error y degradado.
Integracion con /api/v1 esta tipada y validada con mappers por dominio.
Internacionalizacion ES/EN activa en UI, mensajes de error y textos de ayuda.
Eventos de telemetria para acciones clave estan instrumentados con correlation_id.
Accesibilidad base WCAG 2.2 AA en componentes nucleares verificada en QA.
Performance aceptable en escenario de volumen medio con paginacion o virtualizacion aplicada donde corresponda.
4.12 Entregables tecnicos de la fase
El equipo frontend entrega los siguientes artefactos: codigo fuente modular tipado, guia de componentes, especificacion de rutas y estados, matrices de validacion por formulario, contrato de integracion con endpoints backend y tablero de metricas de cliente.

4.8 Ideas UX adicionales (alto impacto, baja complejidad)
- Indicador visual de "precio en tendencia" (flecha arriba/abajo) sin necesidad de grafico complejo.
- Mensaje fijo de "ultima actualizacion" con tono tranquilizador cuando el dato no es fresco.
- Modo comparativo simple: mostrar dos tarjetas lado a lado con precio actual, minimo, ultima variacion.
- Microcopia educativa: explicar que el precio trackeado es orientativo y puede variar en compra final.
- Estado vacio guiado: si no hay vuelos, mostrar un mini tutorial con pasos.
4.9 Ideas avanzadas UX
- Vista "historial compacto": resumen mensual sin grafico completo.
- "Modo decision": pantalla simple que solo muestra si conviene esperar o comprar.
4.10 Consistencia UX obligatoria
- Mostrar en UI: "ultimo precio", "fecha de actualizacion" y estado de frescura.
- Explicar que el precio trackeado es orientativo (base 1 adulto sin extras).
- Busqueda rapida = heuristica (no prometer IA en MVP).
- Identidad de vuelo MVP = Ruta + Dia.
