0. Objetivo y alcance de la Fase 6
Esta fase define el sistema operativo completo de Viru en produccion: como se construye, se despliega, se monitoriza y se recupera ante fallos. El objetivo no es solo "subir una app", sino garantizar continuidad de negocio para un producto que depende de un proveedor externo cambiante y de cargas periodicas de scraping/consulta de precios. Se asume coherencia total con la Fase 3 (backend con adapter de proveedor y uso de ryanair-py) y con la Fase 5 (modelo de datos con crecimiento temporal y retencion controlada).
El alcance cubre entornos, infraestructura base, contenedores, pipelines CI/CD, gestion de secretos, observabilidad, respuesta a incidentes, hardening de seguridad y estrategia de escalado. Se documentan runbooks y criterios de salida a produccion con enfoque enterprise. Toda decision intenta maximizar resiliencia sin sobrecargar de complejidad prematura al equipo.
Se incluye explicitamente una estrategia de gestion de memoria y prevencion de degradacion por recursos, por peticion del Product Owner. Este punto se trata como requisito de calidad y no como mejora opcional. Se aterrizan limites por servicio, politicas de backpressure, deteccion temprana de fugas y mecanismos de recuperacion automatica para mantener la misma calidad en picos y bajo presion.
6.1 Principios DevOps y ADR operacionales
Principio 1: desplegar con seguridad repetible. Ningun cambio entra a produccion sin pasar por pruebas automatizadas, analisis estatico, escaneo de vulnerabilidades y un plan de rollback probado. El pipeline es parte del producto y se versiona junto al codigo.
Principio 2: observabilidad por defecto. Cada endpoint, job y llamada al proveedor se instrumenta desde el inicio con logs estructurados, metricas y trazas distribuidas. Si una incidencia no puede ser observada, se considera deuda tecnica critica.
Principio 3: degradacion controlada. Dado que ryanair-py y el proveedor aguas arriba pueden sufrir cambios o limitaciones, la plataforma debe mantener servicio con respuestas parciales, cache de ultima lectura valida y mensajes de estado transparentes. El fallo externo no puede tumbar toda la experiencia de usuario.
Principio 4: capacidad y costo equilibrados. Se fija un presupuesto de rendimiento y de gasto por entorno. Escalar no significa abrir recursos sin control; significa automatizar limites, detectar saturacion y actuar con reglas de autoscaling y prioridades de cola.
ADR-OPS-01 a ADR-OPS-08
ADR-OPS-01: Contenedores Docker como unidad de despliegue para backend, frontend y workers.
ADR-OPS-02: Orquestacion Kubernetes gestionado (o equivalente PaaS con jobs y autoscaling).
ADR-OPS-03: Pipeline CI/CD con promotion por entornos y aprobacion manual a produccion.
ADR-OPS-04: Observabilidad estandar OpenTelemetry + Prometheus + Grafana + centralizacion de logs.
ADR-OPS-05: Feature flags para activacion gradual de funcionalidades de riesgo.
ADR-OPS-06: Estrategia blue/green o canary para backend; frontend con despliegue atomico.
ADR-OPS-07: Politica de secretos en gestor dedicado; prohibido secretos en repositorio o imagen.
ADR-OPS-08: Politica formal de memoria por servicio con limites, alertas y runbooks OOM.
6.2 Topologia de entornos
Se definen cuatro entornos: local, dev compartido, staging y produccion. Local prioriza velocidad del desarrollador; dev valida integracion entre servicios; staging replica condiciones de produccion para ensayos de despliegue y carga; produccion opera bajo SLO comprometidos. Esta separacion evita mezclar objetivos y reduce el riesgo de errores por configuraciones cruzadas.
El entorno staging no es decorativo: debe tener datos anonimizados representativos, jobs reales en frecuencia reducida y pruebas de resiliencia programadas. Cualquier cambio en provider adapter (incluyendo updates de ryanair-py) se valida primero en staging con pruebas de compatibilidad semantica de respuesta.
La promocion entre entornos usa artefactos inmutables. El mismo container digest que pasa en staging es el que llega a produccion. No se recompila entre pasos para evitar "works in staging, fails in prod".
La configuracion por entorno se gestiona por variables y secretos externos, nunca por ifs dispersos en codigo. El objetivo es que cambiar de entorno sea un cambio declarativo controlado por IaC y no una operacion manual frágil.
6.3 Arquitectura de infraestructura objetivo
Infraestructura base recomendada: cluster Kubernetes gestionado con nodos autoscalables, base de datos PostgreSQL gestionada, Redis gestionado, almacenamiento de objetos para reportes/exports y servicio de colas para eventos asynchronos. La arquitectura puede mapearse a AWS, GCP o Azure sin lock-in funcional porque la capa de despliegue se define con Terraform y Helm.
Servicios principales: API backend, worker de tracking, worker de alertas, scheduler de jobs, frontend web y gateway de observabilidad. El scheduler dispara ventanas de escaneo; los workers ejecutan consultas con control de concurrencia; el backend sirve datos y operaciones del usuario.
El trafico de entrada pasa por CDN + WAF + Ingress. Esto protege frontend y API, reduce latencia y agrega controles de rate limit perimetral. La comunicacion interna usa TLS dentro del cluster cuando el proveedor lo permita y politicas de red para segmentacion minima necesaria.
La base de datos y Redis se consumen via endpoints privados. Se evita exponer servicios de datos a internet publica. Las copias y snapshots se cifran en reposo y en transito. El aislamiento de red se trata como requisito de seguridad operacional, no como opcion.
6.4 Contenerizacion y runtime
Cada servicio se construye con imagen minimalista y proceso unico. Backend y workers comparten base comun para reducir deriva, pero mantienen entrypoints separados para responsabilidades diferentes. Frontend se construye en etapa builder y se sirve por servidor estatico con compresion habilitada.
Las imagenes fijan dependencias por lockfile, incluido ryanair-py, para evitar cambios sorpresivos. Las actualizaciones de librerias se ejecutan por ventana controlada y con pruebas de regresion del adapter de proveedor. Nunca se usa "latest" en tags de imagen ni en dependencias de runtime.
Se definen probes de salud: liveness para detectar procesos colgados, readiness para sacar instancias no aptas del balanceador y startup probe para servicios con warmup. Esto evita reinicios prematuros y reduce falsos positivos de degradacion.
La configuracion runtime incluye timezone UTC, locale estable y limites de archivo/proceso razonables. Para workers con mucha E/S se ajusta pool de conexiones y timeout por tipo de operacion, evitando bloqueos prolongados que consuman memoria de forma acumulativa.

6.5 CI/CD enterprise
El pipeline CI inicia en cada pull request con pasos paralelos: lint, formateo, tests unitarios, tests de contrato API, cobertura minima, escaneo SAST, escaneo de dependencias y build del contenedor. Un PR no mergea si falla cualquier quality gate. Esto protege rama principal y evita deuda invisible.
Tras merge a main se ejecuta build release con versionado semantico, firma de imagen y generacion de SBOM. El artefacto se publica en registro privado y se despliega automaticamente a dev. Luego se promueve a staging con suite de integracion, pruebas de carga corta y smoke tests end-to-end.
La salida a produccion requiere aprobacion manual del responsable on-call o release manager, mas verificacion de ventana de riesgo. El despliegue se hace canary (5%, 25%, 50%, 100%) o blue/green segun tipo de cambio. Si se detecta degradacion en SLI definidos, rollback automatico al revision anterior.
Para cambios de provider adapter o upgrade de ryanair-py se añade etapa especifica de "provider compatibility suite": compara respuestas basicas esperadas, tasas de error y tiempo medio de consulta. Si la desviacion supera umbral, no hay promotion a produccion.
Quality gates recomendados
Cobertura minima backend: 80% lineas y 70% ramas en modulos core.
Cobertura minima frontend: 75% componentes criticos y flujos de auth.
Vulnerabilidades High/Critical: 0 abiertas en imagen release.
Latency budget en smoke staging: p95 API publica < 450 ms en carga nominal.
Error budget previo a release: servicio dentro de SLO en ultimas 24 horas.
6.6 Gestion de configuracion y secretos
La configuracion se divide en tres capas: defaults de codigo seguros, variables por entorno y secretos en gestor dedicado (Vault/Secrets Manager/Key Vault). Las claves de correo, JWT signing keys, credenciales de base de datos y tokens de terceros nunca se inyectan desde archivos en repositorio.
Las rotaciones de secretos son automatas y con doble ventana de validez para evitar caidas por corte abrupto. El backend soporta key rotation de JWT mediante keyset activo/pasivo. Las conexiones a DB y Redis se reciclan de forma gradual cuando rota una credencial.
Se aplican politicas de minimo privilegio: cada servicio tiene identidad propia (service account) y solo permisos necesarios. Ejemplo: frontend no accede a DB; worker-alerting no puede modificar tablas de autenticacion; scheduler solo opera colas y endpoints internos de control.
Toda configuracion sensible queda auditada en logs de control con quien, cuando y que cambio realizo. Se excluye el valor del secreto en logs, mostrando solo metadata. Esta trazabilidad es esencial para incidentes de seguridad y cumplimiento operativo.
6.7 Observabilidad integral (logs, metricas, trazas)
Logs: formato JSON estructurado con campos obligatorios (timestamp_utc, level, service, env, trace_id, user_id_hash opcional, watch_id opcional, event_type). Se prohíben logs multilinea no parseables en produccion. Se define retencion por criticidad: aplicacion 30 dias, seguridad 90 dias, auditoria 180 dias.
Metricas: se instrumentan SLI tecnicos y de negocio. Tecnicos: latency p50/p95/p99, error rate, throughput, saturation CPU/RAM, cola pendiente, tiempo de job. Negocio: precio detectado por ventana, alertas emitidas, alertas suprimidas por anti-spam, ratio de deep-link exitoso.
Trazas distribuidas: backend y workers propagan contexto para diagnosticar flujos completos desde accion usuario hasta captura proveedor y persistencia de snapshot. Esto permite aislar si un problema reside en API, DB, cola o provider.
Dashboards y alertas: se definen tableros por dominio (API, Tracking, Alerting, DB, Infra). Alertas por umbral y por anomalia. Todas las alertas tienen runbook asociado; alerta sin runbook se considera incompleta y no puede quedar habilitada en produccion.

6.8 Operacion de jobs de tracking y alertas
El scheduler define ventanas de escaneo configurables por zona horaria de negocio, con jitter aleatorio para evitar patrones detectables y reducir riesgo de bloqueo externo. Cada planificacion genera tareas idempotentes en cola; la idempotencia evita duplicados al reintentar.
Los workers consumen con control de concurrencia adaptativo: si la tasa de error del proveedor sube, se reduce automaticamente el paralelismo y aumenta el backoff exponencial con jitter. Esta estrategia protege tanto al proveedor como a nuestra estabilidad interna.
Se definen colas separadas por prioridad: critical refresh manual, normal periodic scan y low-priority recomputation/analytics. El refresh manual del usuario no debe quedar bloqueado por batch masivos. Ademas, se implementa dead-letter queue con politicas de reintento escalonado.
El pipeline de alertas evalua reglas tras persistir snapshots. Para evitar ruido se aplica deduplicacion temporal, umbral minimo de variacion y enfriamiento por regla. Se registra motivo de supresion para que soporte pueda explicar por que una alerta no se envio.
6.9 Seguridad operativa y hardening
Se adopta modelo Zero Trust basico: autenticacion mutua entre servicios cuando sea viable, politicas de red restrictivas, imagenes firmadas y escaneadas, y ejecucion de contenedores sin privilegios root. Se deshabilitan capacidades Linux no necesarias y filesystem root en modo read-only.
El ingress aplica rate limiting, proteccion contra ataques comunes y reglas WAF. Para endpoints de autenticacion y sugerencias se agregan controles anti abuso (captcha/rate-limits progresivos). Se protege la API frente a enumeracion de recursos y brute force.
La cadena de suministro de software se protege con SBOM, escaneo continuo de CVE y politica de parcheo con SLA por severidad. CVE critico expuesto a internet: parche < 24h. CVE high: parche < 72h. Cualquier excepcion requiere riesgo aceptado formalmente.
La auditoria de seguridad incluye pruebas de penetracion periodicas, revision de secretos huérfanos y simulacros de fuga de credenciales. El objetivo es convertir seguridad en practica operativa continua, no evento puntual pre-lanzamiento.
6.10 Estrategia de despliegue, rollback y cambios
Se establecen dos modos de release: canary para backend/workers y atomico para frontend. Canary reduce impacto al exponer solo una fraccion de trafico. Durante la ventana canary se monitorean SLI de forma estricta; desviaciones activan rollback sin esperar aprobacion manual.
El rollback es un primer ciudadano: cada release guarda manifiestos, imagenes y migraciones asociadas. Para cambios de esquema se aplica principio expand-contract: primero compatibilidad hacia delante, luego corte de codigo antiguo y por ultimo limpieza. Nunca se mezclan cambios destructivos sin doble compatibilidad.
Se usa feature flags para separar despliegue de activacion funcional. Esto permite entregar codigo inactivo, validar estabilidad y habilitar progresivamente por cohortes de usuarios. Si una feature falla, se desactiva flag sin redeploy.
El calendario de cambios evita ventanas de alto riesgo de negocio. En periodos de demanda alta se restringen cambios no criticos y se prioriza estabilidad. Se mantiene politica de freeze temporal cuando el error budget mensual se consume de forma acelerada.
6.11 Gestion de memoria y estabilidad bajo presion
Este apartado responde directamente al requisito del Product Owner de no degradar calidad por limitaciones de memoria. La estrategia combina prevencion, deteccion y recuperacion. Cada servicio define memory request y memory limit explicitos; operar sin limites declarados queda prohibido en produccion.
Presupuestos iniciales propuestos: api-backend request 512Mi limit 1Gi; worker-tracking request 768Mi limit 1.5Gi; worker-alerting request 384Mi limit 768Mi; scheduler request 256Mi limit 512Mi; frontend request 128Mi limit 256Mi. Estos valores se calibran con test de carga y profiling real, pero sirven como guardrail desde el dia uno.
Se implementa control de backpressure para evitar crecimiento no acotado de memoria: limite de items en batch, streaming en lectura de resultados, paginacion obligatoria en consultas de historico y rechazo controlado cuando la cola supera umbral critico. Si el sistema no puede absorber mas carga, debe degradar elegantemente antes de caer.
Deteccion temprana: metricas de RSS, heap, GC pauses y tasa de reinicios OOMKilled por servicio. Se alerta en tres niveles: warning al 70% del limite, high al 85%, critical al 95% sostenido. Cada alerta incluye recomendacion concreta (escalar horizontal, bajar concurrencia, drenar cola, activar modo safe).
Recuperacion automatica: cuando worker-tracking supera umbral de memoria y error rate simultaneo, se reduce concurrencia dinamicamente y se replanifican tareas no criticas. Si persiste, se reinicia pod de forma escalonada para evitar caida total. El sistema debe preferir progreso parcial estable frente a throughput maximo inestable.
Prevencion de fugas: pruebas de soak de 6-12 horas en staging, heap snapshots en puntos de carga y comparacion de lineas base por release. Cualquier tendencia ascendente sin liberacion consistente se bloquea antes de produccion. Esta regla evita que pequeños leaks se conviertan en incidentes recurrentes.
Para Python runtime se recomienda: limites de tamaño en estructuras en memoria, uso de iteradores/generadores para procesamiento incremental, y limpieza explicita de objetos pesados en loops largos. El adapter de proveedor debe normalizar solo campos necesarios y evitar retener payloads completos cuando no aportan valor.
Runbook OOM resumido
Paso 1: confirmar OOMKilled y servicio afectado en dashboard de infraestructura.
Paso 2: activar flag de concurrencia reducida para workers o rate-limit temporal en API.
Paso 3: drenar cola low-priority y preservar solo tareas critical/normal.
Paso 4: escalar horizontalmente si hay capacidad de nodo disponible.
Paso 5: analizar ultimo release, diff de dependencias y heap profile; decidir rollback si procede.
Paso 6: documentar RCA en menos de 24h con accion preventiva verificable.
6.12 Continuidad de negocio, backup y disaster recovery
Se define RPO/RTO por componente: DB principal RPO <= 5 minutos y RTO <= 30 minutos; Redis RPO <= 15 minutos; artefactos de deployment RPO ~ 0 por registro replicado. Los objetivos se validan en simulacros trimestrales y no solo en papel.
Backups de DB: full diario, incrementales frecuentes y retencion estratificada (7 diarios, 4 semanales, 6 mensuales). Se prueban restauraciones parciales y completas en entorno aislado. Backup sin restore probado no se considera valido.
DR regional: opcion activa-pasiva con infraestructura predefinida por IaC. Ante caida regional, se promueve entorno secundario con procedimientos automables. El plan incluye comunicacion externa, degradacion temporal de features no esenciales y priorizacion de servicios core.
Se mantiene inventario de dependencias criticas (correo, proveedor vuelos, DNS, CDN, DB). Para cada una se documenta plan B y limite de tolerancia. El objetivo es reducir puntos unicos de fallo y acelerar toma de decisiones en incidentes de alta presion.
6.13 Coste, capacidad y FinOps operativo
La gestion de costos se integra en el ciclo de release. Cada nueva funcionalidad debe declarar impacto esperado en CPU, memoria, almacenamiento, trafico y terceros. Sin estimacion de costo no se aprueba rollout global.
Se aplican politicas de autoscaling con topes maximos para evitar crecimiento descontrolado. En horas valle se reducen replicas de servicios no criticos. Para jobs batch se usan ventanas economicas cuando no comprometan SLO de usuario.
El almacenamiento de snapshots crece rapido; por eso se aplica retencion por antiguedad y compresion de datos historicos frios. Las consultas frecuentes se resuelven con agregados/materializaciones para reducir carga de DB y costo de compute.
Mensualmente se revisa tablero FinOps con costo por servicio, costo por usuario activo y costo por 1,000 snapshots capturados. Esta visibilidad permite optimizar con criterio y no por intuicion.
6.14 Gobierno operacional y modelo de guardias
Se establece on-call rotativo con cobertura clara, niveles de severidad y protocolo de escalado. Incidentes Sev-1 activan canal dedicado y war room. La comunicacion debe ser factual, con tiempos y acciones verificables.
Cada alerta productiva tiene owner definido. Las alertas ruidosas se corrigen o se eliminan: ruido sostenido degrada la capacidad real de respuesta. Se mide indicador de "alert fatigue" para mantener eficacia operativa.
Postmortems sin culpa: cada incidente significativo produce RCA con causas tecnicas y organizativas, acciones correctivas y fecha compromiso. El cierre del incidente exige verificar que la accion quedo desplegada y monitorizada.
El gobierno de cambios incluye CAB ligero para releases de alto impacto. No bloquea velocidad; la organiza. El criterio es proteger confiabilidad del producto sin frenar evolucion necesaria.
6.15 Checklist de salida a produccion
Infraestructura IaC aplicada y versionada; drift check en estado limpio.
Pipelines CI/CD en verde con quality gates obligatorios.
Secretos cargados en gestor y rotacion probada.
Dashboards y alertas activas con runbooks asociados.
Pruebas de carga y soak completadas sin OOM ni degradacion grave.
Plan de rollback validado en staging para release actual.
Backups y restore de DB verificados durante la semana de salida.
Feature flags configuradas para activacion gradual.
SLO y error budget publicados y aceptados por PO y equipo tecnico.
Soporte operativo informado con playbook de incidencias frecuentes.
La salida se considera aprobada solo cuando todos los items estan completos y evidenciados. Si falta uno critico, se pospone release. Esta disciplina evita que la urgencia del calendario comprometa calidad y reputacion.

6.18 Cierre de fase y criterios de aceptacion
La Fase 6 se cierra cuando la plataforma puede desplegarse de forma repetible, observable y reversible, con evidencia de estabilidad bajo carga nominal y picos razonables. No basta con "funciona en mi maquina"; debe existir trazabilidad de build, despliegue y operacion end-to-end.
Criterio de aceptacion tecnico: pruebas de humo y de carga en staging superadas; sin vulnerabilidades criticas abiertas; sin alertas sin owner; sin componentes core sin limites de memoria. Criterio de aceptacion de negocio: capacidad de reaccion ante incidencias sin afectar de forma prolongada al usuario final.
Este documento deja base suficiente para ejecutar operacion enterprise con calidad sostenida. La siguiente fase puede profundizar en integracion de mejoras o testing avanzado, pero la columna vertebral DevOps queda definida y lista para implementacion incremental.
Nota de coherencia: ver FASE 1 secciones 1.8 (Contexto de mercado y oportunidad) y 1.9 (Monetizacion y escalabilidad) para alinear decisiones de producto, arquitectura, roadmap y operacion.
6.15 Ideas operativas adicionales (enfoque negocio)
- Alarma de "frescura": porcentaje de vuelos con datos recientes.
- Alarma de "alertas ignoradas": si muchas alertas no se abren, ajustar umbrales.
- Modo "apagado suave": si proveedor falla, el sistema entra en modo solo lectura.
6.16 Ideas avanzadas operativas
- Metrica de "confianza de datos": si el proveedor es inestable, bajar intensidad de alertas.
- Canal interno de incidentes: checklist simple para fallos de proveedor.
