FASE 3 - BACKENDViru - Ryanair Tracker
Documento tecnico - Arquitectura backend, APIs, datos, seguridad y casos limiteFecha: 2026-02-12
0. Objetivo y alcance de la Fase 3
Esta fase define el backend completo del producto Viru a nivel de ingenieria senior: lenguaje, framework, estructura de proyecto, contratos de API, modelo de dominio, seguridad, estrategia de ingesta y control de edge cases. El objetivo no es solo describir componentes, sino fijar decisiones implementables que permitan construir el sistema sin huecos de interpretacion.
El alcance de esta fase se centra en la capa servidor y su comportamiento operativo para el caso de uso principal: tracking de precios de Ryanair, busqueda avanzada, alertas y persistencia historica por usuario. Se mantiene coherencia con la Fase 1 y con las decisiones de arquitectura global de Fase 2, priorizando trazabilidad entre requisito y diseño tecnico.
Como condicion adicional del Product Owner, esta fase incorpora explicitamente la libreria ryanair-py como pieza del proveedor de datos. No se delega la arquitectura al paquete; se integra como dependencia encapsulada mediante una capa de abstraccion que protege al sistema ante cambios de API, latencia y posibles degradaciones externas.
Este documento adopta un enfoque de whitepaper tecnico: cada subseccion incluye decisiones, justificacion, riesgos, trade-offs y plan de mitigacion. Donde existan ambiguedades funcionales heredadas de fase previa, se propone una implementacion por defecto y una ruta de evolucion sin romper contratos publicos.
3.1 Lenguaje, framework y paradigma de backend
Decision D-BE-001: el backend se implementa en Python 3.12 sobre FastAPI. La razon principal es la alineacion con la dependencia ryanair-py y la necesidad de combinar endpoints HTTP de baja latencia con tareas asincronas de rastreo. FastAPI ofrece tipado fuerte con Pydantic, validacion declarativa y especificacion OpenAPI nativa, lo que reduce deuda de contrato desde el primer sprint.
Decision D-BE-002: arquitectura hexagonal (ports and adapters) con capas separadas: API, Application, Domain, Infrastructure. El dominio no depende de FastAPI ni de SQLAlchemy. Esta separacion permite cambiar la fuente de datos de vuelos, sustituir el scheduler o migrar la base de datos sin reescribir reglas de negocio.
Decision D-BE-003: doble modo de ejecucion. El primer modo es un API service para login, consulta, alta de vuelos y accion manual de refresh. El segundo modo es un worker service para trabajos periodicos de tracking, evaluacion de alertas, retries y notificaciones. Esta separacion evita bloquear el plano de atencion al usuario por cargas batch.
Decision D-BE-004: modelo de concurrencia mixto. Las peticiones HTTP usan async I/O donde aporte valor; las tareas intensivas de red se despachan a cola de trabajos con control de concurrencia por usuario y por ruta. El objetivo es limitar fan-out, evitar tormentas de llamadas al proveedor y mantener tiempos de respuesta estables.
Decision D-BE-005: versionado semantico de API desde el inicio (/api/v1). Esto habilita cambios futuros en contratos sin romper clientes web o mobile. Todos los endpoints de Fase 3 se documentan como contrato estable y se protegen con pruebas de compatibilidad de esquema.
3.2 Integracion de ryanair-py dentro del backend
La libreria ryanair-py se usa como adaptador primario de lectura para obtener vuelos one-way y return en ventanas de fechas, con inicializacion por moneda en la instancia del cliente. El paquete documenta instalacion por pip, metodos para cheapest flights y return flights, y salida en objetos tipo Flight/Trip. Esta informacion condiciona el diseño de nuestro ProviderAdapter.
La integracion se define con una interfaz interna IFlightProvider. El dominio nunca llama directamente a ryanair-py. En su lugar, la capa Infrastructure implementa RyanairPyProviderAdapter con metodos normalizados: search_one_way_min(), search_return_min(), normalize_currency(), map_airport(). Esta estrategia permite un fallback futuro (otro proveedor o endpoint propio) sin tocar casos de uso.
Decision D-BE-006: normalizacion agresiva de salida. Aunque ryanair-py permita configurar currency, la propia documentacion advierte que no siempre se respeta. Por eso cada respuesta se persiste con currency detectada, tasa de conversion aplicada (si procede), fuente y timestamp de captura. No se asume homogeneidad de moneda en agregaciones historicas.
Decision D-BE-007: cumplimiento y riesgo legal operativo. La documentacion del paquete incluye disclaimer expreso de no afiliacion con Ryanair y responsabilidad del usuario sobre terminos de uso. Por ello, Viru incorpora controles de frecuencia, cache, backoff y limite por cuenta para reducir agresividad de consulta y operar en modo responsable.
Decision D-BE-008: degradacion controlada. Si el adapter falla por timeout, cambios de contrato o respuesta vacia, el backend no devuelve error destructivo al usuario final. Se responde con ultimo snapshot consistente, bandera stale_data=true y mensaje de transparencia. Paralelamente se encola reintento con politica exponencial y circuito de proteccion por ruta.
Decision D-BE-009: telemetria del proveedor. Cada llamada a ryanair-py genera metricas de latencia, ratio de acierto, ratio de no-disponibilidad, y mismatch de moneda. Estos datos alimentan decisiones de producto (frecuencia de rastreo) y de operaciones (necesidad de fallback o feature flag).

3.4 APIs completas del backend (contrato /api/v1)
El contrato API se diseña bajo principios REST con recursos estables, payloads JSON y errores homogéneos. Todos los endpoints protegidos requieren Bearer JWT. Se aplican limites por usuario y por IP para impedir abuso y proteger los integradores externos.
Formato de error estandar:
{ "error": { "code": "ALERT_RULE_INVALID", "message": "Regla invalida", "details": {...}, "correlation_id": "..." } }
Politica de idempotencia: POST sensibles aceptan encabezado Idempotency-Key. Si la misma clave se repite en ventana activa, se devuelve el mismo resultado persistido. Esto evita duplicados por reintentos de red.
3.4.1 Auth y sesion
POST /auth/register crea usuario con email y password robusta, envia verificacion y retorna estado pending_verification. POST /auth/login autentica y entrega access_token corto + refresh_token rotatorio. POST /auth/refresh invalida refresh anterior y emite uno nuevo para evitar replay.
POST /auth/logout revoca refresh token activo del dispositivo. POST /auth/forgot-password inicia flujo de recuperacion con token de un solo uso y TTL corto. POST /auth/reset-password consume token y fuerza cierre de sesiones antiguas.
GET /auth/me devuelve perfil, idioma, timezone y estado de verificacion. Este endpoint se usa en bootstrap del frontend para hidratar preferencias y mostrar UX contextual.
3.4.2 Watchlist y tracking de vuelos
POST /watchlist crea un vuelo a vigilar con origen IATA, destino IATA, fecha de salida, precio_objetivo opcional, preferencias de escaneo y politica de alertas. Valida fecha futura, origen distinto de destino y limite maximo de watches por plan.
GET /watchlist lista vuelos del usuario con paginacion cursor-based y filtros por estado (active, expired, archived). GET /watchlist/{watch_id} devuelve detalle con ultimo precio, moneda, tendencia corta y health de proveedor.
PATCH /watchlist/{watch_id} permite actualizar precio objetivo, activar o pausar seguimiento, y ajustar filtros sin perder historial. DELETE /watchlist/{watch_id} hace soft-delete para preservar auditabilidad y evitar perdida de trazabilidad en alertas enviadas.
POST /watchlist/{watch_id}/refresh-now ejecuta consulta inmediata al proveedor y persiste snapshot si existe cambio material o si no hay muestra previa para el periodo actual. El endpoint responde 202 Accepted cuando se delega a cola y 200 cuando se completa inline bajo SLA.
POST /watchlist/refresh-bulk permite refresco manual de multiples watch_id con limite de lote y backpressure. Se evita fan-out no controlado mediante colas priorizadas y scheduler de ventanas.
3.4.3 Historico de precios y analitica
GET /prices/history recibe watch_id y rango temporal. Retorna serie temporal normalizada con puntos [timestamp, price, currency, source_quality]. Incluye indicadores min, max, avg, last y delta_pct para soportar grafico y calendario.
GET /prices/calendar compacta los datos por dia para render de calendario. Incluye flags is_daily_min y is_daily_max para coloreado inmediato en frontend sin procesamiento adicional.
GET /prices/compare acepta lista de watch_id (2..N) y retorna dataset armonizado por timezone del usuario. Si hay monedas mixtas, la respuesta reporta currency_mode='mixed' y opcionalmente conversion de referencia segun tasa del dia.
GET /prices/export genera CSV o JSON firmado temporalmente para descarga asincrona. Esta API es util para usuarios avanzados y para soporte interno.
3.4.4 Busqueda rapida, alternativos y escalas
POST /search/quick ejecuta busqueda sobre origen, destino, fecha y parametros flexibles (radio_km, dias_antes, dias_despues, incluir_escalas, filtros horarios). Este endpoint no crea watch automaticamente; solo devuelve oportunidades y scoring.
El backend evalua aeropuertos alternativos cercanos al origen y destino usando indice geoespacial y politicas de distancia por defecto del perfil. Las combinaciones se rankean por costo total, desviacion temporal y distancia terrestre estimada.
Si incluir_escalas=true, el motor genera propuestas self-connect de dos tramos con verificacion de buffer minimo de conexion. La respuesta incorpora risk_label (low/medium/high), minutos_de_buffer y disclaimer explicito de conexion no protegida.
POST /search/save-result transforma un resultado de busqueda en watchlist activo en una sola accion idempotente. Esto reduce friccion de UX y mantiene trazabilidad desde consulta hasta seguimiento.
3.4.5 Alertas, notificaciones y preferencias
POST /alerts/rules crea reglas por watch: threshold_below, threshold_above, notify_on_every_change, cooldown_minutes, min_change_pct. GET/ PATCH /alerts/rules/{id} gestiona ciclo de vida de reglas y su prioridad.
GET /notifications lista eventos enviados (email, in_app, push cuando aplique) con estado delivered, bounced, failed. Permite auditoria de soporte y transparencia al usuario sobre por que una alerta no llego.
PATCH /preferences define idioma, moneda preferida, filtros persistentes y ventanas horarias deseadas. Estas preferencias alimentan defaults de busqueda y politicas del scheduler.
POST /feedback/suggestions acepta propuestas del usuario con anti-spam, captcha y sanitizacion. Se almacena con metadatos de cliente para triage interno, sin exponer datos sensibles en respuestas publicas.
3.5 Modelo de datos backend (dominio y persistencia)
El modelo de datos se diseña para preservar historico completo sin sacrificar rendimiento de consulta. Se distingue entre entidades transaccionales (usuarios, watchlist, reglas) y series temporales de alta cardinalidad (price_snapshots).
Entidad User: id UUID, email unico, password_hash, is_verified, locale, timezone, created_at, updated_at, deleted_at. Invariante: email canonizado en minusculas y bloqueo temporal tras intentos fallidos consecutivos.
Entidad FlightWatch: id UUID, user_id, origin_iata, destination_iata, travel_date_local, status, target_price, is_paused, scan_policy_json, created_at. Invariante: combinacion user_id + origin + destination + travel_date + deleted_at(NULL) puede limitarse por politica para evitar duplicados no intencionales.
Entidad PriceSnapshot: id ULID, watch_id, captured_at_utc, departure_time_local opcional, raw_price, raw_currency, normalized_price_eur opcional, provider='ryanair-py', provider_payload_hash, is_stale, quality_score. Invariante: unique(watch_id, captured_at_utc, provider_payload_hash) para deduplicacion fuerte.
Entidad AlertRule: id UUID, watch_id, type, threshold_value, cooldown_minutes, min_change_pct, enabled, last_triggered_at. Entidad NotificationEvent: id UUID, rule_id, channel, status, sent_at, provider_message_id, retry_count.
Entidad UserPreference: id UUID, user_id, default_radius_km, avoid_departure_before, language, currency_pref, include_stops_default, updated_at. Entidad Suggestion: id UUID, user_id opcional, text, source_ip_hash, spam_score, created_at.
Para rendimiento, se recomiendan indices compuestos: (watch_id, captured_at_utc DESC) en snapshots; (user_id, status, travel_date_local) en watchlist; (rule_id, sent_at DESC) en notifications. Para volumen alto, particion temporal mensual en snapshots.
3.6 Orquestacion de tareas backend y consistencia
El backend incorpora scheduler para escaneos periodicos y cola para trabajos distribuidos. El scheduler determina que watches deben consultarse segun politicas de frecuencia, proximidad a fecha de vuelo, volatilidad reciente y carga de sistema.
Flujo de trabajo principal: seleccionar watches activos -> agrupar por ruta y fecha para minimizar llamadas -> consultar proveedor -> normalizar respuesta -> persistir snapshots -> evaluar reglas -> emitir notificaciones. Cada etapa genera eventos de auditoria con correlation_id.
Decision D-BE-013: exactitud eventual con garantias de no duplicado. El sistema no promete actualizacion instantanea absoluta, pero si garantiza que un mismo snapshot no se procesa dos veces y que una misma alerta no se dispara fuera de politica de cooldown.
Decision D-BE-014: locking optimista con compare-and-set en estados sensibles (ejemplo: refresh manual concurrente con escaneo programado). Si se detecta carrera, se conserva la escritura con timestamp mas reciente y se registra conflicto para observabilidad.
Decision D-BE-015: backoff y circuit breaker por proveedor. Tras errores consecutivos por ruta o region, el circuito se abre temporalmente y los trabajos entran en cola diferida. Se evita degradar toda la plataforma por fallo externo localizado.
3.7 Seguridad, autenticacion y proteccion de datos
La seguridad backend se construye por capas. Capa 1: autenticacion JWT con access token corto y refresh rotatorio con revocacion por dispositivo. Capa 2: autorizacion por propietario de recurso (user_id) en cada endpoint de negocio. Capa 3: protecciones de abuso y de entrada.
Credenciales: password hash con Argon2id, politica de longitud minima y bloqueo escalonado por intentos fallidos. Verificacion de email obligatoria para activar alertas externas. Reset de password invalida sesiones previas y tokens activos.
Protecciones API: rate limit por IP y por usuario, validacion estricta de IATA, fechas y rango de parametros, saneamiento de strings en sugerencias, y rechazo de payloads sobredimensionados. CORS restringido por entorno, HSTS, y cookies seguras cuando aplique.
Proteccion de datos: cifrado en reposo para campos sensibles, cifrado en transito TLS, minimizacion de PII en logs, y retention policy configurable. Los logs tecnicos nunca deben almacenar passwords, tokens completos ni contenido libre del usuario sin mascarado.
Decision D-BE-016: trazabilidad sin sobreexposicion. Se registran IDs tecnicos y hashes para diagnostico, pero no datos personales en claro salvo campos estrictamente necesarios y con acceso limitado por rol interno.
Compliance operativo: dado que el proveedor subyacente no es oficial de la aerolinea, se añade en backend un banner de transparencia y terminos de uso internos. Se documenta que la disponibilidad de datos puede variar y que las recomendaciones no son garantia de precio futuro.
3.8 Edge cases, riesgos y mitigaciones backend
Caso limite E-001 - Moneda inesperada: el proveedor puede devolver una moneda distinta a la solicitada. Mitigacion: almacenar moneda real, marcar inconsistencia, convertir de forma opcional para analitica y evitar comparaciones silenciosas incorrectas.
Caso limite E-002 - Ruta sin disponibilidad temporal: puede no haber vuelos para rango dado. Mitigacion: persistir snapshot de ausencia (no-data snapshot) con razon, para diferenciar entre 'sin vuelos' y 'fallo tecnico'.
Caso limite E-003 - Cambios bruscos y alerta spam: oscilaciones cortas pueden disparar muchas notificaciones. Mitigacion: cooldown por regla, umbral minimo de variacion y politica digest configurable.
Caso limite E-004 - Fecha local vs UTC: diferencias de zona horaria pueden mover vuelos de dia en frontera nocturna. Mitigacion: almacenar travel_date en timezone del aeropuerto y captured_at en UTC; convertir solo en lectura.
Caso limite E-005 - Refresh manual simultaneo mas scheduler: riesgo de duplicado o estado inconsistente. Mitigacion: idempotency key, lock por watch y deduplicacion por hash de payload.
Caso limite E-006 - Degradacion del proveedor: respuestas lentas o estructura cambiante. Mitigacion: timeouts cortos, retries con jitter, circuit breaker y fallback a ultimo dato confiable.
Caso limite E-007 - Uso malicioso de endpoints de busqueda: consultas masivas para scraping inverso. Mitigacion: rate limit adaptativo, deteccion de patrones anormales, cuota por plan y challenge progresivo.
Caso limite E-008 - Data drift para prediccion futura: historico insuficiente o sesgado por estacionalidad. Mitigacion: score de confianza, no emitir recomendacion fuerte por debajo de umbral de evidencia y exponer razon de abstencion.
3.9 Decision record resumido (ADR de backend)
ADR-BE-01: Python + FastAPI por ecosistema, velocidad de iteracion y compatibilidad con ryanair-py.
ADR-BE-02: Arquitectura hexagonal para desacoplar dominio de proveedores externos y framework web.
ADR-BE-03: Integracion de ryanair-py solo a traves de adapter con normalizacion de moneda y manejo de errores.
ADR-BE-04: Persistencia principal en PostgreSQL y cache/cola con Redis para trabajos y throttling.
ADR-BE-05: Modelo de alertas con cooldown, min_change_pct e historial de envios para anti-spam.
ADR-BE-06: Seguridad con JWT rotatorio, Argon2id, rate limiting y minimizacion de PII en logs.
ADR-BE-07: API versionada /v1 y contratos estables con pruebas de esquema y idempotencia en POST sensibles.
ADR-BE-08: Degradacion elegante ante fallo proveedor: stale snapshot + reintento asincrono + transparencia de estado.

3.11 Ideas adicionales backend (no tecnicas para PO)
- Registro de "intentos de alerta" para entender si el usuario ignora alertas.
- Logica de "ventana de calma" para no repetir alertas similares en poco tiempo.
- Modo degradado: si el proveedor falla, el backend devuelve el ultimo dato con mensaje explicito.
- Endpoints de "resumen" (minimo, maximo, ultimo cambio) para evitar cargar historico completo en cada vista.
3.12 Ideas avanzadas backend (impacto en negocio)
- Endpoint de "impacto de alertas": medir si el usuario hizo click tras alerta.
- "cooldown inteligente" en alertas basado en respuesta previa del usuario.
3.13 Reglas canonicas aplicadas
- Precio trackeado = base 1 adulto sin extras.
- Identidad de vuelo = Ruta + Dia en MVP.
- Alertas MVP = email + in-app.
Cierre de la Fase 3
Con este documento queda especificada la implementacion backend de nivel empresa para Viru, incluyendo el uso responsable de ryanair-py y un diseño robusto frente a cambios externos. La siguiente fase podra consumir estos contratos y decisiones para construir frontend y workflows de usuario sin ambiguedad tecnica.
